# Sonar v0.3.0

## Summary

Sonar v0.3.0 consolidates Blackglass resilient retrieval into Sonar and improves
the MCP surface used by agent runtimes.

## Highlights

- existing Sonar HTTP retrieval remains the default fast path;
- difficult HTML can optionally escalate through Scrapling HTTP and
  CloakBrowser;
- policy and robots denials are terminal and never trigger fallback;
- fetched bodies and retrieval provenance persist in SQLite;
- fetch followed by extract reuses the cached body;
- PDFs and other non-HTML documents never trigger browser fallback;
- HTTP responses and prepared bundles add retrieval provenance fields;
- raw MCP names are service-local, preventing `sonar_sonar_*` names in OpenCode;
- MCP exposes `scrape` as the simple one-call tool for a known URL;
- MCP supports stdio and Streamable HTTP;
- lightweight deployments do not require browser dependencies.

## Operations

- `sonar-mcp` keeps stdio as its default transport and supports Streamable HTTP
  through `SONAR_MCP_TRANSPORT=streamable-http`.
- The default Docker target remains lightweight. Build `browser-runtime` for
  the optional Scrapling, CloakBrowser, and Playwright runtime.
- Existing SQLite databases migrate additively on startup. Cached bodies and
  provenance use new nullable/defaulted columns; no destructive migration is
  required.
- `force_refresh=true` bypasses cached extraction and retrieval bodies.

## Breaking Change

Raw MCP tool names changed from `sonar_*` to unprefixed service-local names such
as `search` and `extract`. Clients that namespace tools with a connection named
`sonar` continue to display the intended `sonar_search` and `sonar_extract`
names.
